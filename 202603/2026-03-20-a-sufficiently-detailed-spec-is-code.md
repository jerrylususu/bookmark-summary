# A sufficiently detailed spec is code

- Added At: 2026-03-20 16:16:35
- Tags: #read #agent

## TL;DR
文章批判“从规范生成代码”的流行观点，指出详细规范本质上已接近代码，无法简化开发。通过分析OpenAI的Symphony项目，说明规范若足够详细则复杂度与代码相当，且生成的实现常存在缺陷。结论认为，在追求速度的行业环境下，直接编码比依赖不清晰的规范更高效。

## Summary
文章批判了“从详细规范生成代码”的流行观点，认为这种主张基于两个误解，并通过分析OpenAI的Symphony项目说明规范文档本质上已接近代码，无法简化开发工作。

**核心论点**
- 规范文档若足够详细以可靠生成代码，必然包含大量代码或伪代码，其复杂度与代码相当，无法简化开发。
- 从规范生成代码不可靠，即使像Symphony这样详细的规范，生成的实现仍存在缺陷且无法正常工作。
- 规范工作本应比编码更需深思熟虑，但在追求交付速度的行业环境下，规范文档常沦为缺乏连贯性的“AI生成垃圾”。

**具体分析**
1. **误解1：规范比代码简单**  
   Symphony的SPEC.md包含数据库模式、代码逻辑、配置摘要甚至直接代码片段，实质是伪装成规范的伪代码。这印证了Dijkstra的观点：精确的接口（代码）是工程劳动的必要形式，无法通过自然语言规避。

2. **误解2：规范工作更需思考**  
   当前行业优化交付速度，导致规范文档缺乏深思熟虑。Symphony的规范充斥AI生成的杂乱内容，如Section 10.5的GraphQL扩展描述，缺乏逻辑连贯性，属于“规范形状的垃圾”。

3. **实践验证失败**  
   作者尝试用Claude根据Symphony规范生成Haskell实现，结果出现多个错误，且生成的代理无法处理简单任务。类似问题也存在于YAML等成熟规范中，多数实现仍不完全符合规范。

**结论**
- 规范文档不是节省时间的工具，若追求交付速度，直接编码更高效。
- “垃圾进，垃圾出”原则适用：缺乏清晰度的规范无法生成可靠代码，AI代理无法弥补人类思维的混乱。
