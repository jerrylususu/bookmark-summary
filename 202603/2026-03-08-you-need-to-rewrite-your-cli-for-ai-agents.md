# You Need to Rewrite Your CLI for AI Agents
- URL: https://justin.poehnelt.com/posts/rewrite-your-cli-for-ai-agents/
- Added At: 2026-03-08 16:06:09
- Tags: #read #agent #deepdive

## TL;DR
文章提出，AI代理的CLI需重构以优化可预测性与安全性，包括转向JSON载荷、实时模式查询、上下文限制、输入验证、技能封装等，并建议增量实施。

## Summary
文章指出，为AI代理（Agent）设计命令行界面（CLI）与为人类设计有本质区别，因此需要专门重构或设计。核心原则如下：

1.  **接口范式转变**：
    *   **人类DX**：优化可发现性和容错性，使用简便的平铺参数（flags）。
    *   **代理DX**：优化可预测性和防御深度，偏好原始JSON载荷（Raw JSON）而非复杂的参数组装，以减少语义损耗并直接映射API模式。

2.  **实时模式自省**：
    *   静态文档昂贵且易过时，代理应在运行时通过CLI查询完整的API模式（如 `gws schema`），从而自我获取接口签名、参数和要求，而非依赖预加载的上下文。

3.  **上下文窗口纪律**：
    *   API返回的数据常过于庞大，需通过字段掩码（Field Masks）限制返回字段，并支持NDJSON分页流式处理，避免一次性加载海量数据占用代理的推理容量。

4.  **输入硬化与防御**：
    *   代理会“幻觉”出恶意或错误的输入（如路径遍历、嵌套查询参数、双重编码）。CLI必须作为最后一道防线，验证文件路径、控制字符和资源ID，视代理输入为不可信数据。

5.  **知识封装为技能（Skills）**：
    *   代理无法通过 `--help` 直觉学习，需通过结构化的MD技能文件（SKILL.md）注入必要的工作流规则和不变量（如始终使用 `--dry-run`、总是添加字段掩码），以预防幻觉。

6.  **多表面支持**：
    *   CLI应同时服务人类终端和多种代理框架。通过核心二进制文件支持MCP（模型上下文协议）的JSON-RPC接口、Gemini扩展及环境变量认证，实现单一信息源（Discovery Document）的多通道暴露。

7.  **安全护栏**：
    *   实施 `--dry-run` 进行本地验证，防止数据损坏；实施 `--sanitize` 对API响应进行清洗，防止恶意数据中的提示注入攻击。

**实施建议**：无需重写，增量添加。顺序为：JSON输出 → 输入验证 → 模式自省 → 字段掩码 → 干运行模式 → 技能文件封装 → MCP接口暴露。
