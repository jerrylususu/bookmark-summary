## TL;DR
Google API密钥安全模型变化导致风险：原本用于公共服务的密钥被静默赋予访问敏感Gemini端点的能力，可能引发数据泄露和费用激增。建议开发者检查并审计密钥，避免公开暴露。

## Summary
这篇文章揭示了Google API密钥安全模型的一个重大变化及其带来的风险。核心问题在于，Google长期向开发者宣传其API密钥（如用于Maps、Firebase等的`AIza...`格式密钥）并非秘密，可以安全地嵌入客户端代码。然而，当Gemini API推出后，这些原本用于公共服务的密钥被静默地赋予了访问敏感Gemini端点的能力，而开发者并未收到任何警告。

**主要问题：**
1.  **追溯性权限扩展**：为公共服务（如Google Maps）创建并公开部署的API密钥，在项目启用Gemini API后，自动获得了访问私有数据（如上传文件、缓存内容）和产生LLM使用费用的能力。
2.  **不安全的默认设置**：新创建的API密钥默认为“无限制”，可立即用于项目中所有已启用的API（包括Gemini），这违背了安全设计原则。

**影响与发现：**
*   攻击者只需从公开网页（如网站源代码）中抓取这些密钥，即可访问敏感端点，导致数据泄露、账单激增或服务配额耗尽。
*   研究团队通过扫描2025年11月的Common Crawl数据集，发现了2,863个暴露在公网上的此类密钥，受害者包括大型金融机构、安全公司，甚至Google自身。Google自己的一个长期公开密钥也被证实可以访问其内部Gemini服务。

**披露与修复时间线：**
*   报告于2025年11月21日提交，Google最初认为这是预期行为。
*   在提供Google自身密钥的证据后，Google于12月2日将问题重新分类为“漏洞”，并开始采取措施（如限制已泄露密钥的访问）。
*   Google承诺修复根本原因，并在其路线图中提出改进措施，包括为新密钥设置更严格的默认范围、主动拦截已泄露密钥以及加强通知。

**给用户的建议：**
1.  **检查项目**：登录GCP控制台，检查每个项目是否启用了“Generative Language API”。
2.  **审计密钥**：如果已启用，检查API密钥配置，确认是否有无限制或明确允许Gemini API的密钥。
3.  **验证公开性**：确认这些密钥是否暴露在公共位置（如前端代码、公共仓库）。如有，立即轮换密钥。
4.  **使用工具扫描**：可使用TruffleHog等工具扫描代码和资产，查找泄露的Google API密钥并验证其是否具有Gemini访问权限。

**结论：**
这一事件暴露了在现有平台上集成AI功能时，遗留凭证可能意外扩大攻击面的问题。尽管Google已采取行动，但根本性的架构问题（单一密钥格式用于不同目的）仍需解决。开发者应主动审计和保护其API密钥，即使它们曾被认为是“非秘密”的。
