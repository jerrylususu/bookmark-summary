# Production query plans without production data
- URL: https://boringsql.com/posts/portable-stats/
- Added At: 2026-03-10 13:45:29
- Tags: #read #database

## TL;DR
PostgreSQL 18 引入便携式优化器统计信息功能，通过 `pg_restore_relation_stats` 和 `pg_restore_attribute_stats` 函数，允许在测试环境中注入生产级统计信息，从而在不需要实际数据的情况下模拟真实查询计划，提升CI/CD测试与本地调试效率。

## Summary
本文介绍了 PostgreSQL 18 引入的便携式优化器统计信息功能，允许在不依赖生产数据的情况下部署生产级查询计划。核心内容如下：

### 问题背景
- 计划 CI/本地测试环境的数据库统计信息与生产环境差异巨大，导致 `EXPLAIN` 结果无法反映真实生产计划，影响回归测试和调试效果。
- 传统方式依赖 `ANALYZE` 获取统计信息，但需要实际数据，耗时且无法跨环境共享。

### 核心功能
1. **`pg_restore_relation_stats`**  
   - 直接向 `pg_class` 注入表级统计信息（如行数、页数）。
   - 示例：通过注入 5000 万行表的统计信息，可改变优化器对表规模的感知，但仅调整行数不会改变计划（因列级统计仍缺失）。

2. **`pg_restore_attribute_stats`**  
   - 向 `pg_statistic` 注入列级统计信息（如直方图、MCV 列表、相关性）。
   - 示例：  
     - 为 `created_at` 列注入跨 5 年的直方图，使优化器选择索引扫描而非全表扫描。  
     - 为 `status` 列注入倾斜分布（95% 为 "delivered"），使优化器对 "pending" 查询使用位图索引扫描，而对 "delivered" 使用顺序扫描。  
   - 关键点：优化器会根据实际文件大小缩放注入的行数估计值，但保持比例不变，确保计划形状正确。

### 操作流程
- **导出统计信息**：使用 `pg_dump --statistics-only` 生成 SQL 文件，包含 `pg_restore_*` 函数调用。
- **导入统计信息**：在目标数据库（如测试环境）执行导出的 SQL，无需生产数据。
- **完整工作流示例**：  
  1. 导出生产环境架构和统计信息。  
  2. 在测试环境创建数据库并加载架构。  
  3. 注入统计信息（可搭配最小化测试数据）。  
  4. 此时测试环境的查询计划与生产环境一致。

### 注意事项与限制
- **统计信息持久化**：自动清理（autovacuum）会覆盖注入的统计信息，需通过 `ALTER TABLE` 禁用分析或设置极高阈值（仅适用于只读测试场景）。
- **局限性**：  
  - 无法完全模拟生产数据量（优化器会根据实际文件大小调整估计值）。  
  - PostgreSQL 18 不支持扩展统计信息（如多变量相关性），需等待 PostgreSQL 19 的 `pg_restore_extended_stats()`。
- **安全**：需 `MAINTAIN` 权限（可通过 `GRANT pg_maintain` 授权），避免使用超级用户。

### 应用场景
- **CI/CD 测试**：在测试环境中模拟生产统计信息，提高回归测试可靠性。
- **本地调试**：重现生产环境的慢查询计划，无需连接生产数据库。
- **版本升级**：PostgreSQL 18 通过该功能在主版本升级时自动迁移统计信息，避免长时间 `ANALYZE`。

### 总结
PostgreSQL 18 的便携式统计信息功能将优化器统计转化为可部署的轻量级构件（通常 <1MB），使开发、测试环境能精准模拟生产查询计划，大幅提升调试和测试效率。但需注意统计信息持久化和扩展统计的局限性，且严禁在生产环境直接使用。
