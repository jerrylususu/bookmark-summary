## Summary
本文系统总结了Claude Code的架构、治理与工程实践，核心围绕六层模型展开：

1. **六层架构模型**  
   - CLAUDE.md / rules / memory：长期上下文，定义项目契约  
   - Tools / MCP：动作能力，扩展Claude的操作范围  
   - Skills：按需加载的工作流与方法论  
   - Hooks：强制执行规则的拦截层  
   - Subagents：隔离上下文的自治工作单元  
   - Verifiers：验证闭环，确保结果可信可审计  

2. **上下文工程**  
   - 上下文成本构成：固定开销（系统指令、工具定义等）占200K上下文的15-20%，动态可用部分约160-180K  
   - 最佳实践：保持CLAUDE.md简短可执行，按需加载Skills，使用Subagents隔离探索任务，避免上下文污染  
   - 压缩机制陷阱：默认压缩可能丢弃架构决策，需在CLAUDE.md中明确Compact Instructions  

3. **Skills设计**  
   - 按需加载，描述符需明确触发条件，避免过长正文  
   - 三种类型：检查清单型、工作流型、领域专家型  
   - 反模式：描述过短、正文过长、一个Skill覆盖多任务  

4. **工具设计原则**  
   - 好工具：名称清晰、参数明确、返回信息直接相关、边界清楚  
   - 坏工具：名称模糊、参数泛化、返回噪声多  
   - 演进教训：独立工具（如AskUserQuestion）比参数或格式约定更稳定  

5. **Hooks应用**  
   - 适合：阻断危险操作、自动格式化、动态上下文注入  
   - 不适合：复杂语义判断、长时间运行流程  
   - 与CLAUDE.md、Skills三层叠加，确保规则执行  

6. **Subagents使用**  
   - 核心价值是隔离而非并行，适合探索、测试等产生大量输出的任务  
   - 配置时需显式约束工具、模型、最大轮次，避免权限过宽  

7. **Prompt Caching架构影响**  
   - 缓存按前缀匹配，顺序重要：系统提示→工具定义→对话历史→当前输入  
   - 避免在静态内容中加入动态信息，会话中途勿切换模型  
   - Compaction通过fork上下文实现，保持缓存命中率  

8. **验证闭环**  
   - 分层验证：命令退出码、集成测试、生产监控  
   - 在Prompt、Skill、CLAUDE.md中明确定义验收标准  

9. **高频命令工程意义**  
   - 上下文管理：/context、/clear、/compact  
   - 能力治理：/mcp、/hooks、/permissions  
   - 会话连续性：--continue、--fork、-p非交互模式  

10. **CLAUDE.md编写指南**  
    - 应放：构建命令、架构边界、代码约束、禁止事项  
    - 不应放：大段背景、空泛原则、低频知识  
    - 让Claude维护自身规则，定期review更新  

11. **实践经验**  
    - 环境透明化：通过doctor命令统一状态报告  
    - 混合语言项目：Hooks按文件类型触发检查  
    - 完整工程布局：解耦全局约束、路径规则、工作流与架构细节  

12. **常见反模式与修复**  
    - CLAUDE.md当wiki、Skill大杂烩、工具过多、无验证闭环等  
    - 修复方向：拆分上下文、明确职责、绑定验证器、最小化自治范围  

13. **配置健康检查**  
    - 开源工具claude-health可一键检查配置状态，输出优先级报告  

14. **结语**  
    - 三阶段演进：工具使用者→流程优化者→系统设计者  
    - 核心问题：明确任务完成标准，否则不适合自主完成
