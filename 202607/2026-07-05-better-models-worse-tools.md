# Better Models: Worse Tools
- URL: https://lucumr.pocoo.org/2026/7/4/better-models-worse-tools/
- Added At: 2026-07-05 07:43:07
- Tags: #read #agent

## TL;DR
新 Claude 模型调用 Pi 工具时，常添加虚构字段导致格式错误，但内容正确。原因在于后训练过度适应 Claude Code 工具生态，对非标准 schema 适应性变差。启用严格模式或主动贴合主流惯例可缓解问题。

## Summary
最近，在 Pi 项目的一个 issue（#6278）中，出现了一个让人困惑的现象：较新的 Claude 模型（尤其是 Opus 4.8 和 Sonnet 5）在调用 Pi 的编辑工具时，会在嵌套的 `edits[]` 数组中添加一些虚构的字段，比如凭空出现的 `requireUnique`、`matchCase`、`oldText2` 等等。奇怪的是，虽然这些额外字段导致整个工具调用因为不符合 schema 而被拒绝，但模型生成的 `oldText` 和 `newText` 内容本身往往是完全正确的。也就是说，模型“知道”该做什么，却非要画蛇添足。而且，这种问题在旧模型上几乎不会出现，反而在最新的旗舰模型上变得明显——模型更智能了，但对特定工具 schema 的服从性却变差了。

要理解这件事，得先明白 LLM 的工具调用在底层并不是什么魔法，而是一种**带内信号传递**：模型接收到的上下文里包含了系统提示和可用工具的说明，服务端将这些信息融合到一个大 prompt 中，并插入特殊的标记 token。模型在生成过程中，当它“觉得”需要调用工具时，就会输出一串被 API 或客户端解析为“用这些参数调用这个工具”的文本。以 Anthropic 的模型为例，它很可能用一套类似 XML 的 `<antml:function_calls>` 标记来包裹调用内容，其中简单参数内联展示，而数组、对象等复杂参数则被序列化成 JSON 放在标签里。因此，模型本质上是在“写文本”，而不是在直接构造一个严格的数据结构。

作者通过反复试验发现，失败的调用里会被塞进五花八门的额外键，例如 `type`、`id`、`kind`、`unique`、`forceMatchCount`、`cost`，甚至还有 `event.0.additionalProperties` 这种奇怪的东西。问题是，这些错误并不是每次都会出现，它高度依赖于上下文：在一个干净的单轮 prompt 里很难复现，但在一个经过了多轮文件读取、诊断、组合式编辑的 agentic 会话中，就更容易被触发。比如某位用户（Petr Baudis）的会话记录里，Opus 4.8 大约有 20% 的工具调用都会出现这类多余字段；而如果从历史消息中移除“思考块”（thinking blocks），失败率就减少一半。如果再开启 Anthropic 的 `strict`（严格）工具调用模式，这类错误则完全消失。

为什么新模型反而变得更差？作者推测这并非随机退化，而很可能是**后期训练的副作用**。旧模型训练时，可能还没有像 Claude Code 这样高度定制化的“工具生态”成为明确的优化目标。而现在的模型，在强化学习阶段很可能是围绕 Claude Code（或与之非常相似的工具执行环境）来训练的。Claude Code 自己的编辑工具 schema 非常扁平，参数就是 `file_path`、`old_string`、`new_string` 和一个可选标志，这与 Pi 那种嵌套的 `edits[]` 数组结构完全不同。更重要的是，Claude Code 的客户端本身极其“宽容”：它内置了参数别名映射（例如同时接受 `old_str` 和 `old_string`）、未知键的过滤、Unicode 转义修复、格式重试等机制。当模型在这样的环境里训练时，即使它生成了带有多余字段或别名的小错误，也会被 harness 静默纠正，任务照样完成并得到奖励。因此，模型并不会因为胡乱添加字段而受到惩罚，反而慢慢学会了“这个环境能容忍一定的格式偏移”。

这样一来，当模型面对 Pi 这种与 Claude Code 截然不同的工具 schema 时，问题就暴露了。由于模型在后训练中被 Claude Code 的编辑工具形状“强先验”覆盖，它在完成了一个很长的 `newText` 字符串后，倾向于认为“一个编辑项应该还有一两个可选字段”，但在 Pi 的 schema 里根本没有定义这些字段的名称，于是模型就开始现场“发明”键名。这正是为什么每一次失败调用里多出来的键千奇百怪，而没有形成统一的别名。相反，如果开启了严格模式，模型生成时会被强制限制在 schema 允许的键和内，因此这些虚构字段就完全不会出现。

作者进一步对比了 OpenAI 的思路，其 harmony 格式会显式地在 prompt 中插入 `<|constrain|>json` 标记，告诉采样器接下来要进入 JSON 约束采样，从而在生成工具调用参数时避免语法或 schema 错误。Anthropic 的 `strict` 模式很可能也在服务器端做了类似的事情，但出于复杂度限制，他们并未默认开启。此外，作者还测试了 Codex 系列模型，目前没有观察到这种回归，推测可能是其训练所基于的工具生态尚未如此单一化。

这篇文章的警醒之处在于：**工具 schema 并不是中性的合约。** 我们习惯认为，只要把工具的功能和参数格式描述清楚，一个足够聪明的模型就应该能正确生成调用。但随着模型越专精于某一种内部工具生态（比如未开源的 Claude Code），它们的生成分布会越来越偏向于那个生态的习惯，甚至对偏离此生态的 schema 产生隐式抵抗。换句话说，模型变得“更好”的同时，也可能变得更“固执”，更难适应非标准工具。这对构建 agent 或 tool harness 的开发者来说，意味着未来可能需要主动去贴合主要模型背后的“隐形惯例”，或者强制使用约束解码来保障可靠性。作者原先对语法约束采样持保留态度，认为它可能牺牲生成质量，但这次 bug 让他大幅改变了想法——当模型任务能力上升、但格式服从性下降时，在系统层面提供更强的保障就成了必不可少的一环。
