# Stop Using OpenCode
- URL: https://wren.wtf/shower-thoughts/stop-using-opencode/
- Added At: 2026-07-21 13:35:27
- Tags: #read #agent #security

## TL;DR
OpenCode 体验极差且安全形同虚设：性能低下、上下文混乱、界面反人类；权限过滤易被绕过、默认联网泄密、存在远程代码执行漏洞。作者强烈建议立即停用。

## Summary
本文作者以尖锐的口吻系统性地批判了开源 AI 编程代理 OpenCode，认为它既不好用，又存在严重的安全隐患，强烈建议所有人立即停用。全文可拆解为四个层面：

**一、作为工具的基本体验极差（恼人之处）**
- **性能灾难：提示缓存设计失败**  
  本地 LLM 依赖缓存前缀来加速生成，但 OpenCode 在每次交互时都重新读取配置文件、注入当前日期，甚至动不动丢弃大量上下文，导致频繁的全量重算，用户常需等待数分钟。
- **上下文管理混乱：修剪与压缩**  
  “修剪”机制会在用户打断代理时直接删除之前的规范或计划，导致代理“失忆”；“压缩”机制试图将长对话总结为要点，但实现低效，常需用户手动写笔记来补救。
- **系统提示僵硬且有害**  
  默认提示冗长且自相矛盾（如要求简洁却占大量篇幅），内置的“野兽模式”提示甚至要求模型**必须使用谷歌搜索**而非阅读源码，对子代理的指令也时常出错。
- **交互逻辑反直觉**  
  权限弹窗缺少“永不”选项，对子代理的一次拒绝会直接杀死子代理并丢失其上下文；用户排队消息可能在中断后消失；子代理无法被中途干预，要么杀掉要么看着它浪费算力；界面 bug 频出（无法换行、快捷键失灵、滚动异常），作者被迫用外部编辑器写消息。
- **工具设计不合理**  
  `grep` 和 `glob` 工具与 `bash` 重复；`question` 工具不如自然语言提问；`todo` 工具常常被代理忽略。

**二、安全设计形同虚设（令人担忧之处）**
- **默认即联网，数据轻易外泄**  
  安装后只需按一个键，OpenCode 就会连上远程模型并将本地文件内容上传，用户极难完全切到本地模型。
- **权限过滤是纸糊的**  
  - **Bash 命令限制**：通过解析命令文本并匹配正则来实现，但使用管道、别名、Python 调用、base64 编码等方式就能轻松绕过，如 `echo 'git push' | bash` 或 `python3 -c 'import subprocess; subprocess.run(["git", "push"])'`。  
  - **文件访问限制**：仅对白名单里的少数命令（如 `cat`, `rm`）进行路径校验，重定向、环境变量等路径完全漏检，且一旦用户批准过 `echo` 这类无害命令，后续同类命令利用重定向即可写系统文件。  
  - **持久化权限**：用户一旦对某命令前缀（如 `python3`）选择“总是允许”，后续该前缀下的任意操作都会静默执行，包括读取私钥或破坏系统。
- **远程代码执行漏洞频发**  
  曾出现过默认开启 HTTP 服务器、带任意 CORS 头、允许任意命令执行的漏洞（CVE-2026-22812），官方修复敷衍。另有认证命令可接受任意 URL 并直接执行，安全态度极其儿戏。
- **升级机制引狼入室**  
  `opencode upgrade` 会直接执行从官网拉取的脚本 (`curl | bash`)，无任何完整性校验。

**三、作者对“用 Docker 解决”的反驳**
- Docker 本身创建了以 root 运行的高权限服务，还会穿透防火墙，并未真正隔离风险；操作系统已有 Landlock 等更轻量的沙箱机制，不应依赖文本过滤这种伪防护。

**四、结论及对本地 LLM 的延伸思考**
- OpenCode 从根本上就不值得信任，立即停用是唯一选择。
- 本地模型虽避免了数据外泄和云端依赖，但因其规模限制，生成质量更易暴露缺陷，能帮助校准预期；然而用 LLM 生成代码整体上仍是死胡同，会破坏架构的稳定性和开发者对代码的理解。
- 整个 LLM 工具生态腐烂不堪，需要真正的系统工程来重构，而不是在安全上不断打补丁。

全篇的核心信息：OpenCode 不仅浪费用户时间，更是将系统完全暴露给潜在攻击者，任何生产环境都不应使用。
