## TL;DR
这篇文章主张用Embedding、BM25等传统算法替代大模型做日常信息匹配，只在定义标准和最终总结时才调用LLM，从而大幅降低Token成本，高效实现个性化筛选。

## Summary
这篇文章讲的是如何用“古法 AI”来省钱高效地做个性化信息筛选，避免什么事都丢给昂贵的大语言模型（LLM），从而省下大量 Token。

现在很多 AI 信息系统，本质上就是把一篇篇文章喂给 LLM，让它判断“这篇和我关心的主题相关吗？”。如果你每天看 500 篇文章，关心 5 个主题，最简单粗暴的做法要调用 2500 次判断，成本直接是文章数乘以主题数。作者认为，这只把人工阅读的成本转移成了模型调用的成本，算不得真正的智能。

其实很多任务并不需要“理解”，只需要计算“距离”。Embedding（嵌入）可以把一段文本转换成一串数字（向量），代表它在语义空间中的位置。两段话意思越接近，向量的方向就越相似，计算余弦相似度就能量化这种距离。所以，判断一篇文章是否和“科技向善”有关，只需要把文章转成向量，再和几个事先存好的主题范文向量比比相似度就行。文章向量算一次就能跟所有主题比较，主题范文向量还能长期缓存；这样成本就从“文章数 × 主题数”降到了接近“文章数 × 1 次 Embedding”，用一个本地小模型就能跑，甚至不用花 API 的钱。

这还不算完全抛弃 LLM。LLM 更适合“定义标准”而不是天天在流水线上执行标准。当用户只给出一个模糊的主题名（比如“科技向善”四个字）时，可以让 LLM 一次性生成几条正向范文（应该被选中的文章长什么样）和负向范文（看起来像但应该排除的），之后把这些范文向量化存起来，日常筛选就再也不需要 LLM 了。只有最后需要把筛出来的少量文章写成综述时，才再请 LLM 出场。

处理去重和聚类，传统方法照样好用。例如用 n-gram（把标题切成连续的字符片段）来计算两篇文章的标题相似度，花销近乎为零，而且能准确认出“腾讯发布混元”和“腾讯正式发布混元大模型”是同一件事。先用 Embedding 做语义召回，再用 n-gram 做精确确认，最后才把合并过的事件簇交给 LLM 去写总结，LLM 只需处理十几个小簇，而不是几百篇原始文章。

如果用户连主题都懒得给呢？作者的“邸报”项目更进一步，不要求你事先定义兴趣。它根据你收藏、点赞、读完、隐藏、不感兴趣等行为，自动学习出多个“兴趣向量”（兴趣簇）。收藏的文章被 Embedding 后，会和现有兴趣簇比较：够近就加权更新这个簇的中心，太远且信号强烈就新建一个簇。正负兴趣分开维护，信号强度也分档（误点不会立刻带偏画像），还会随时间衰减、合并和裁剪。这样一来，你的兴趣就像是一张会被行为不断修正的“语义地图”。

有了这张地图，再结合经典的 BM25 关键词检索（用于抓取专有名词、缩写和新词，避免 Embedding 的盲区），以及 MMR（最大边际相关性，避免排出来 20 篇全是相似内容），就能生成一份个性推荐列表。系统甚至用了一个轻量的在线学习算法 FTRL 来微调各个特征（如语义相似度、关键词、新鲜度、来源偏好）对个人的权重，且初期只在暗处学习，积累足够数据后才谨慎地参与排序。

整篇文章的核心省钱哲学是：让大模型退出主循环，只在定义标准、最终概括等低频或高价值环节出场。日常的高频匹配与排序，完全交给 Embedding、BM25、n-gram、MMR 这些“古法”技术。

作者还特意点明了开头“今日头条”那个例子的用意——那是写作上的钩子，因为头条级的大规模推荐主要靠协同过滤（利用海量用户行为互相推荐），而个人日报没有那么多用户，反而恰恰适合用上一代推荐系统里那些内容匹配、去重、多样性控制等“边角技术”。在个人信息流这个尺度上，这些稳定、便宜、可解释的旧技术完全够用，犯不着一上来就调用 LLM。这也给了现在直接从 LLM 入门编程的开发者一个提醒：省 Token 的关键不在模型大小，而在架构设计，有些智能早在 AI 大爆发之前就存在了，值得补课。
