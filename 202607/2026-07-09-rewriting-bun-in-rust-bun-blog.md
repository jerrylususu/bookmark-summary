## TL;DR
Bun团队利用Claude AI在11天内将50余万行Zig代码重写为Rust，通过全部测试并根除内存安全问题，总费用16.5万美元。

## Summary
Bun 是一个用 Zig 语言编写的高性能 JavaScript 运行时，集成了打包器、包管理器、测试运行器等功能，月下载量超 2200 万。由于 Zig 手动管理内存与 JavaScriptCore 垃圾回收之间的交互极为脆弱，代码中频繁出现 use-after-free、内存泄漏、双重释放等稳定性问题，即使已经采用 Address Sanitizer、模糊测试和大量泄漏检测测试，这类 bug 仍在不断出现。

作者团队曾考虑通过代码风格指南和手工智能指针来加强安全，但最终认为那无法从根本上解决问题。Rust 通过所有权模型和 `Drop` 机制可在编译期杜绝这些内存错误，因此决定将整个 Bun 代码库从 Zig 移植到 Rust。按传统方法，这项 50 多万行代码的重写需要一个小团队花费一整年，而且期间无法发布新功能或修复，这不可接受。

转折点在于作者使用了尚在预发布阶段的 Claude Fable 5 模型，并借助 Claude Code 的动态工作流能力，将整个重写过程压缩到 11 天。具体做法不是简单地“把代码交给 AI”，而是**精心设计了一整套并行流水线**：

1. **准备阶段**：先与 Claude 讨论，生成了详细的移植指南（PORTING.md）和所有结构体字段的生命周期分析表（LIFETIMES.tsv），明确 Zig 模式到 Rust 的映射规则。
2. **对抗性审查（Adversarial Review）**：每个改动都由一个“实现者”Claude 编写，再由两个独立的“审查者”Claude 在隔离的上下文中专门挑错，审查者只看到变更差异，目标就是找出 bug，之后再由一个“修复者”Claude 合并修改。这种机制在合并前就捕获了大量隐患（如异步关闭中的 UAF）。
3. **分阶段并行执行**：
   - 将全部 1448 个 `.zig` 文件分批交给多个 Claude 实例同时机械移植，最多同时运行 64 个实例，峰值提交速度达到每分钟 58 次提交、约 1300 行代码。
   - 所有代码先提交到独立分支，然后运行 `cargo check`，将约 16,000 个编译错误按 crate 分区，再分配 64 个 Claude 并行修复，每个 crate 同样经过实现-双审查-修复循环。
   - 编译通过后，运行 Bun 命令行子命令，修复启动崩溃和运行时错误。
   - 最后执行 Bun 完整的测试套件（包含数百万次断言），继续按测试文件分组，由 Claude 循环修复失败用例，直到全部通过。

整个过程累计消耗约 59 亿 token 输入、6.9 亿 token 输出，API 费用约 16.5 万美元。最终产出超过 100 万行 Rust 代码，所有 4000 多个测试文件全部通过，没有跳过或删除任何测试。重写后约 4% 的代码位于 `unsafe` 块，绝大多数是单行指针操作或调用 C/C++ 库，未来会逐步向更地道的 Rust 演进。

重写中依然引入了 19 个已知回归，它们多源于两种语言在语义上的细微差异（如 `debug_assert!` 在 release 模式会删除内部表达式、切片长度处理差异、边界检查策略不同等），但所有回归已被修复。此后团队还引入了对全部解析器的 24 小时覆盖率引导模糊测试，并让 AI 自动提交修复补丁。

作者坦言，这是当前技术可能性的前沿展现，如果没有这种规模的 AI 辅助，Bun 的 Rust 重写永远不会发生——现实的替代方案只是继续在 Zig 中手工修补那些永无止境的内存问题。
