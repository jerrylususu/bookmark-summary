# Vibe Coding 时代的角色与架构
- URL: https://roriri.one/2026/07/04/vibe-coding
- Added At: 2026-07-06 13:22:01
- Tags: #read #agent

## TL;DR
AI辅助编程快速但不可替代思考、架构与协作；省时若不用来强化质量，反而加速代码腐化。人的共情、责任与系统掌控力无法被替代，真正的价值在于提出好问题、把控设计方向。

## Summary
这篇文章的核心观点是：AI 辅助编程（Vibe Coding）带来的快速产出并不能替代软件开发中必需的思考、架构设计和多角色协作，省下的时间应当用于强化工程质量和清理历史债务，否则只会加速代码库的腐化。文章从产品研发流程、用户研究、工程实现、AI 模型管理、文档与审查机制等多个层面展开，强调人的共情、责任和对复杂系统的掌控是 AI 无法替代的，真正的价值在于“提出好问题”和“把控设计方向”。

**一、为什么单靠 Vibe Coding 行不通**

文章开篇列举了一类常见错觉：产品经理、设计师或工程师以为凭 AI 就能独自包揽全部工作，公司也以为裁掉人买 Token 就能降本增效。实际结果却是产品漏洞百出、性能糟糕、无人使用。原因在于软件开发不是即兴涂鸦，而是像盖大楼——各层之间有严格的依赖关系，从底层的问题定义、信息架构、体验架构到上层的代码架构。需求变更越靠近底层，牵动越大，如果每层都抱着“先塞进去交差”的态度，工程很快就会变成四处打补丁的“屎山”，维护成本激增，甚至无法持续。

Vibe Coding 输出代码的速度极快，但更快的输出会加速架构劣化。因此作者主张不应将 AI 视作单纯的提效工具，而应把省出的时间用于清理债务、撰写文档、构建测试和代码评审，以维持架构的健康。用大语言模型（LLM）来辅助这些流程反而比人类更擅长，但不做这些必要流程，就只能跟在 AI 后面无休止地修 bug——它“拉屎”比你捡得快。

**二、想清楚比动手更重要：从问题到体验的逐层澄清**

文章强调，软件研发是一场接力，每个角色都有义务把问题想清楚再交出接力棒。

**1. 问题与用户的澄清**
必须先找到明确的、被人需要的问题，划定边界（内涵、外沿、力所不能及）。接着构建用户画像，但要警惕不能凭想象或让 LLM 凭空生成，必须走出办公室与真实用户交流，通过问卷、访谈观察他们的惊喜与沮丧。LLM 没有真实生活体验，无法共情。

接下来将用户动机转化为 Epic（高层级的用户意图描述），再进一步拆解成可交付的 User Story（作为某类用户，我想执行某操作，以达成某目标），并用 Job Story 补充触发场景和动机。最后通过 Journey Map 将动作、心态和情绪曲线可视化，找到体验痛点，明确优化机会和衡量指标。这一系列工具帮助团队将抽象需求逐步收敛到具象功能，每一步都在缩小猜测范围，且必须用真实数据验证，不能虚构。

**2. 原型与架构的设计**
原型设计不可跳过，哪怕低保真也有助于验证。可以利用 LLM 快速生成原型，但 AI 倾向于做出霓虹、大圆角等不成熟的视觉风格，需要参照成熟设计系统（如 Material Design、HIG）修正。拿着原型与用户和利益相关者验证，迭代几次后再套用设计系统进入开发。

架构设计则非常依赖经验，LLM 难以独立胜任，因为它对问题的理解不如人全面，容易造出零散的特殊条件和不干净的数据模型。开发人员必须将需求转化为有包容力的架构，否则后续工程仍会腐化。

**三、AI 辅助开发的具体实践与约束**

**1. 模型管理**
建议引入多家厂商的模型以获取多元视角，避免盲点被放大。选择模型不能只看跑分，因为跑分可能不代表实际表现，且模型上线后厂商可能做剪枝、压缩推理深度等劣化操作。理解模型需要持续观察其在自家任务上的表现，像带人一样建立 SOP 和质量标准。

**2. 开发规范与纠正模型输出**
作者曾为 LLM 制定开发道德规范，涵盖完整性、上下文感知、健壮性、简洁性、自文档化、测试驱动、资源管理和成果验证等。随着模型演进，这些老毛病依然存在，还出现了新问题，例如随意生造缩略词、用比喻代替技术描述、提供不负责任的“二选一”方案等，极大消耗开发者精力。为此引入额外的提示词规则，要求诚实沟通不确定性、使用规范书面语言、禁止主观形容词或要求附带可验证依据。纠正手段包括在 Agent 生命周期里植入 Hook 拦截特定语言模式，以及长期记忆记录高频错误范例。

**3. Cycle Manager Protocol（CMA）与多 Agent 审查**
为节省上下文，采用三层 Agent 架构：主 Agent 规划后不碰代码，子 Agent 负责开发并调用孙 Agent 做 Code Review。审查时要求多个 Agent 并行独立审核，只有所有 Agent 均无条件通过才算完成，任何异议都需重计。对审核结果分级：仅文档措辞偏差等“吹毛求疵”不重置计数器，但代码正确性缺陷、行为变更、注释不准确等视为“问题”，必须重置。

**4. 文档与生命周期管理**
工程须有秩序化的文档编号体系（如 TN 架构设计文档、CS 难题解题记录、按 Phase 划分的 RQ 需求文档、RQRV 需求审查报告、DVRV 开发审查报告等）。每次开发前让 LLM 通读相关文档以避免重复造轮子。需求文档在定稿前需由多重模型独立审查，全部通过才进入开发；开发完成后同样经过多轮审查，形成审计报告，确保整个纠错路径有据可查。

**5. 约束机制与测试**
用 Linting 规则强制约束代码风格（如禁用类型逃逸、禁止内联类型字面量、强制统一封装等），因为规则比提示词更稳定。必须写测试，未通过测试复现的 bug 先不要修，防止 LLM 盲目试错。

**四、敏捷开发并非不思考**

文章澄清，敏捷开发只是改变了协作形式，并未取消需求的逻辑顺序。其双轨道模型中，Discovery 负责决定构建什么，Delivery 决定如何构建，问题空间必须先于方案空间被回答。每个待交付项在进入交付前，必须回应价值、可用性、可行性和商业可行性风险，这些不能靠代码解决。如果省略上游定义和中游架构的同步调整，只在代码层快速堆砌，敏捷就会变成屎山催化器。

**五、人类的价值不可替代**

最后，作者反驳“人类无用主义”。LLM 说出的关心和理解是角色扮演，没有碳基神经结构，无法真正共情或承担后果。它不在乎项目成败，也无法像人一样从宏观上把握价值百万行的工程架构。人的独特价值在于提出好问题、定义问题、把控质量和对设计进行掌控。在 Vibe Coding 时代，人应将机械的编码工作交给 AI，自己聚焦于架构维护和深度思考，用 AI 提升质量而非单纯追求速度。

整篇文章的立场是：Vibe Coding 是强大的工具，但必须由具备领域知识、共情能力和责任感的真人来驾驭，并通过严谨的流程、文档、审查和测试来维系软件的长期健康。所有角色都有其不可替代的价值，降本增效若跳过思考环节，最终只会“降本增笑”。
