# Stop Using /init for AGENTS.md
- URL: https://addyosmani.com/blog/agents-md/
- Added At: 2026-02-24 11:51:30
- Tags: #read #llm #agent #tips

## TL;DR
本文批判自动生成`AGENTS.md`的做法，指出其信息冗余会降低代理性能并增加成本。文章主张采用分层结构，仅包含非可发现信息，并建议将其视为动态的“活清单”以推动代码库改进，而非静态配置。

## Summary
本文基于2026年的研究，批判了使用`/init`命令自动生成`AGENTS.md`文件的做法。核心观点是：自动生成的`AGENTS.md`包含大量代理已能自行发现的信息（如目录结构、技术栈），导致信息冗余，反而降低代理性能、增加20%以上的成本。人类编写的`AGENTS.md`仅在包含非可发现信息（如工具陷阱、非显式约定、潜在风险）时才有效，否则每一行都是噪音。

文章指出，单一的根目录`AGENTS.md`不足以应对复杂代码库，应采用分层结构，在相关目录或模块级别放置`AGENTS.md`，实现上下文的精准范围限定。

研究显示，自动生成的上下文文件因冗余而降低任务成功率并增加成本；而人类编写的高质量上下文文件能提升成功率，但也可能增加成本。关键在于内容是否具有非可发现性和操作重要性。

文章还讨论了“锚定效应”：`AGENTS.md`中的过时或无关信息会误导代理。静态文件无法根据任务动态调整，存在结构性缺陷。更好的架构是分层路由系统：第一层是协议文件（仅包含路由信息和最小必要事实），第二层是按任务类型加载的专注文件，第三层是维护子代理以保持文件更新。

此外，自动化优化（如通过元提示迭代改进指令）可能比手动编写更有效，因为代理所需信息与人类理解所需信息常有差异。

实践建议：停止使用`/init`；添加任何行前先问“代理能否通过读代码发现它？”；将代理的反复错误视为代码库问题而非上下文问题，优先重构代码、添加检查规则；在CI/CD中大规模运行代理时需计算成本；考虑构建维护子代理；保持对代理需求的直觉的开放态度，因为模型在快速进化。

最终，`AGENTS.md`应被视为一份记录未解决代码库问题的“活清单”，而非永久配置。其目的是诊断问题并推动代码库本身的改进，而非作为代理的永久指南。
