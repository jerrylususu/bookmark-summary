# Microsoft Has Killed Widgets Six Times. Here's Why They Keep Coming Back.
- URL: https://xakpc.dev/windows-widgets/history/
- Added At: 2026-02-07 15:11:56
- Tags: #read #ui #design #history #deepdive

## TL;DR
Microsoft自1997年起六次尝试在Windows中引入部件功能，均因性能、安全、空间等问题失败。每次重生吸取教训，当前版本优化为轻量、原生渲染和用户可控设计。

## Summary
Microsoft 自 1997 年以来，六次尝试在 Windows 中引入部件（widgets）功能，但每次都以失败告终，随后又不断重生。这一历史呈现出一种循环模式：Microsoft 发布部件功能，用户或攻击者发现其根本缺陷（如性能、安全、屏幕空间、隐私或参与度问题），Microsoft 便紧急终止该功能，并在下一版本中以更严格的约束重建。每个时代的失败教训都塑造了当前部件的架构设计。

以下是各时代的详细总结：

- **推送时代（1997–2001）**：Active Desktop 将 HTML 渲染到桌面上，通过 Channel Bar 推送内容。失败原因是性能问题：处理器和内存资源不足，导致系统崩溃和磁盘 I/O 过载。教训是：环境信息不能以性能为代价。

- **玻璃时代（2007–2009）**：Windows Vista 引入侧边栏部件，展示 Aero Glass 效果。失败原因是屏幕空间问题：固定侧边栏在低分辨率显示器上占用过多空间，用户反感。教训是：部件不能永久占用工作区。

- **自由时代（2009–2012）**：Windows 7 允许部件自由放置在桌面，第三方生态繁荣。失败原因是安全问题：部件无沙盒保护，易受远程代码执行攻击（如 Black Hat 2012 演示），导致 Microsoft 彻底移除该功能。教训是：自由必须伴随沙盒安全，推动了声明式 UI（如 Adaptive Cards）的发展。

- **Metro 时代（2012–2021）**：Windows 8 使用 Live Tiles 作为全屏开始屏幕的部件。失败原因是参与度问题：只读设计不便互动，用户需切换上下文，开发者投入不足。教训是：部件需具互动性，并与工作流程共存。

- **插曲时代（2015–2021）**：包括 Cortana Cards 和 News and Interests，试图提供个性化信息。失败原因是隐私和干扰问题：Cortana 需深度数据访问引发担忧，News and Interests 的悬停触发被视为侵扰。教训是：部件需故意触发，避免侵入性。

- **板时代（2021–2024）**：Windows 11 引入部件板，使用 WebView2 渲染。失败原因是 WebView2 重量和强制新闻源：内存消耗大，用户批评强制内容。教训是：部件需轻量、用户可控。

- **开始体验时代（2024–至今）**：受欧盟数字市场法案驱动，部件板重建为原生 WinUI 3 渲染，新闻源可选，锁屏部件回归。当前状态为活跃发展，吸取了所有历史教训：声明式 JSON 无代码执行（安全）、原生渲染（性能）、覆盖布局（屏幕空间）、互动设计（参与度）、可选数据（隐私）。

总体而言，当前部件平台是历史伤痕的体现，但仍有挑战，如开发者 API 限制和可发现性问题。Microsoft 的部件历史表明，部件应服务于“一瞥即走”的场景，而非与应用程序竞争桌面空间。
