# Building Agents that Don't Break Themselves
- URL: https://fly.io/blog/building-agents-that-dont-break-themselves/
- Added At: 2026-06-18 13:14:40
- Tags: #read #agent

## TL;DR
构建健壮智能体的关键是将决策（大脑）与执行（双手）分离：智能体运行在稳定环境，所有命令在可销毁的临时沙箱中执行。此架构提升安全性、灵活性和容错能力，如Hermes Agent和SpriteDoc案例所示。

## Summary
本文的核心主张是：构建健壮的智能体，关键在于将智能体的“大脑”（长期运行的决策循环）与其“双手”（执行不可信代码的环境）彻底分离。具体做法是让智能体在稳定环境中运行，而将其所有命令执行委托给独立的、可随时销毁和重建的临时沙箱（如 Fly.io 的 Sprite）。文章通过两个实际案例和具体实践要点阐释了这一架构的优势。

**核心理念：大脑与双手分离**
- **智能体本体（大脑）**：可以是一个长期运行的进程（如虚拟机、VPS或笔记本电脑），负责调用模型、维护记忆和状态，无需特殊保护。
- **命令执行（双手）**：所有由模型生成的、可能具有破坏性的代码（如 `bash` 命令）都必须在与智能体隔离的沙箱中运行。沙箱应被视为可随意处置的资源。

**关键优势与实践案例**
1.  **安全性提升**：
    - 沙箱作为安全边界，可保护智能体宿主及连接的其他系统免受错误或恶意命令的破坏。
    - 案例：Hermes Agent 将命令发送到单独的 Sprite 中执行，因此可以省略繁琐的“你确定吗？”二次确认步骤。
2.  **架构灵活性**：同一沙箱技术可以支持不同的生命周期策略。
    - **按会话隔离**：如 SpriteDoc，为每个用户会话启动一个全新的 Sprite，会话结束后销毁，实现多用户隔离。
    - **按任务持久化**：如 Hermes Agent，为每个任务保留一个 Sprite，跨会话复用其安装的环境和文件。
3.  **凭证安全**：
    - 在 SpriteDoc 中，用户的认证令牌仅在命令执行瞬间注入沙箱环境，命令结束后即消失，沙箱本身从不长期存储密钥。
4.  **容错与恢复**：
    - 沙箱的状态可以低成本地设置检查点（Checkpoint）。
    - 当智能体执行了破坏性命令（如删除关键文件）后，只需快速恢复到之前的检查点状态，而无需从备份中恢复整个系统。

**即使智能体自身已沙箱化，仍应执行命令分离**
- 测试表明，即使将智能体本身运行在一个 Sprite 内，它也应该将要执行的命令分派到 *另一个独立的* Sprite 中。这进一步强调了执行环境必须与智能体自身环境物理隔离的原则。

**总结**：要求智能体“小心行事”是徒劳的。正确的做法是将其执行操作的能力限制在一个可以随时丢弃、重建且与核心系统隔离的沙箱环境中。这种“可燃”的执行层设计，使得智能体在安全的前提下，能够大胆尝试并拥有真正的容错能力。
