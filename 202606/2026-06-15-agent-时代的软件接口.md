# Agent 时代的软件接口

- Tags: #read #agent

## TL;DR
本文提出通过DSL与编译器设计，将Agent模糊意图转化为确定性执行，构建稳定软件系统。核心包括策略原语引擎、分层编译管线及验证修复循环，适用于结构化领域，实现Agent工作流的可靠迭代。

## Summary
文章探讨了在Agent时代如何构建稳定、可靠的软件系统，核心范式是通过领域特定语言（DSL）和编译器设计，将Agent的模糊意图转化为确定性执行。作者通过量化投资系统（策引）和Minecraft建筑项目（MinePilot）的实践，总结出以下架构原则：

1. **问题背景**：直接让大模型在复杂环境（如游戏）中实时控制动作容易因误差累积而失败，需寻找更稳定的范式。
2. **核心转变**：从指令式控制转向生成高层蓝图（如.schem文件），但底层数据仍易出错；因此引入“策略原语引擎”范式，提供受限的DSL空间供Agent表达。
3. **DSL设计**：收拢底层控制权，为Agent搭建“语义脚手架”。例如，在量化系统中用原语（如EMA、逻辑判断）组合策略；在MinePilot中用ComponentPlan DSL描述建筑组件，避免直接处理坐标计算。
4. **编译器架构**：系统像编译器一样工作，将DSL转换为中间表示（IR/DAG），实现静态分析和确定性执行。分层管线包括：意图输入、领域计划、中间表示（DAG）、目标表示（如体素计划）。
5. **验证与修复循环**：报错设计为机器可读的契约（JSON格式），提供精确错误定位和修复提示，使Agent能迭代修正。事前通过契约（如LLM_AUTHORING_CONTRACT.md）约束生成，事后通过精准反馈形成修复循环。
6. **职责切割**：采用混合Agent架构，本地Agent处理模糊意图（如自然语言交互），远端处理器处理确定性任务（如编译执行），避免概率性与确定性层混合。
7. **范式边界**：该范式适用于有明确结构、可抽象原语、可验证中间状态的领域（如金融、游戏），但不适用于高度开放创意任务。验证不等于绝对正确，需人类最终决策。
8. **软件架构反转**：传统软件设计从界面到数据库，而Agent时代先设计DSL和编译器内核，再构建交互界面。作者称此为“Domain-shaped, not Domain-locked”范式。

总结：通过DSL、编译器和验证契约，将Agent工作流结构化，实现稳定迭代，是下一代软件架构的关键。
