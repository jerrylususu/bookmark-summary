# One Human + One Agent = One Browser From Scratch

## TL;DR
本文记录作者与LLM代理在三天内合作开发跨平台浏览器的经历，支持HTML/CSS渲染，代码超2万行。实验表明一人一代理模式高效，协作质量比代理数量更重要。

## Summary
本文记录了作者结合一个人类（自己）和一个LLM代理（如Codex）在约三天时间内从头开始构建一个基本浏览器的经历。浏览器支持HTML和CSS渲染（不支持JavaScript），并能在Windows、macOS和Linux上运行。

**开发背景和约束**：作者设定严格要求，包括三天内完成、禁止使用第三方Rust库、仅依赖操作系统自带工具、支持多平台、能渲染特定网站（如个人博客和Hacker News），且代码必须可编译和可读。目标是验证人与代理协作的效率。

**开发过程**：
- **第一天**：从基础功能起步，如渲染“Hello World”和嵌套标签，添加截图功能用于测试，并集成X11和cURL进行网页获取和渲染。代码增至约7500行，文件均小于1000行。
- **第二天**：优化体验，添加headless模式以避免测试窗口干扰，修复窗口大小、兼容性和性能问题，改进字体渲染。工作流由人类选择网站，代理复制渲染结果。
- **第三天**：扩展功能，支持滚动、调试日志和后退按钮，并添加macOS和Windows支持，确保跨平台兼容。
- **第四天**：进行收尾工作，修复持续集成（CI）问题，发布最终版本。总耗时控制在72小时内。

**成果**：最终代码库约20,150行，涵盖布局、样式、平台适配等模块，成功实现基本浏览器功能，代码结构清晰。项目开源，可供编译和测试。

**心得**：
- 一人一代理的模式比一人使用多个代理更高效，代理能专注在单一代码库上取得实质进展。
- 此模式可扩展到多人各带代理，潜力巨大。
- 开发中“慢即是快”，注重质量胜过速度。
- 人类在驱动代理过程中的角色可能比代理技术本身更关键。

**结论**：实验表明，少量资源（一人一代理）在短时间内能完成复杂项目，暗示单纯增加代理数量并非提升自动化编码效率的可靠方式，而人与代理的协作质量更为重要。
