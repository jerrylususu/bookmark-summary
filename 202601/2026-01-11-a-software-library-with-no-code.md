## TL;DR
作者提出“无代码”软件库（如whenwords），仅依赖规格说明和测试用例，通过AI自动生成代码。这种方法适合简单工具，但复杂、高性能或需社区支持的场景仍需传统代码库。AI或改变编码方式，但代码实现和社区价值不会消失。

## Summary
作者发布了一个名为“whenwords”的无代码软件库，它不包含具体代码，而是基于规格说明（SPEC.md）和测试用例（tests.yaml），通过AI工具（如Claude）自动生成多种语言（如Ruby、Python等）的实现。该库提供五个简单功能，如将时间戳转换为人类可读字符串。

作者以此探讨AI编码代理能力提升后，软件工程的可能变化。核心观点包括：

- **无代码库的优势**：对于简单工具库，无代码方式可以实现语言无关性，降低维护成本。
- **适用范围**：适合简单、标准化的工具（如whenwords），但不适用于复杂场景。
- **需要传统代码库的情况**：
  1. **性能关键**：如浏览器等需要高度优化的场景。
  2. **测试复杂**：多语言、多AI代理的测试矩阵会迅速扩大，难以管理。
  3. **支持与调试**：无代码库难以复现和修复特定环境下的bug。
  4. **持续更新**：需要安全补丁或快速迭代的库（如LiteLLM、Postgres）。
  5. **社区与互操作性**：开源社区带来的协作、测试和长期维护是关键价值。

结论是，无代码库可能适用于一次性工具，但复杂、高性能或依赖社区的软件仍需要传统代码库。未来随着AI进步，这一平衡可能改变，但社区和代码实现的价值不会消失。
