## TL;DR
AI代理生成的PR激增，审查需警惕技术债务。建议结构化流程：检查CI、代码重复、关键路径逻辑，并借助工具辅助，平衡效率与质量。

## Summary
文章指出，随着AI代理生成的代码拉取请求（PR）数量激增，传统的人工审查流程面临挑战。代理生成的代码表面整洁但可能隐藏技术债务，审查者需保持警惕并采取结构化方法。

**核心问题与现状**
- 代理PR已占GitHub代码审查的20%以上，审查带宽被饱和。
- 研究显示代理代码冗余度更高，但审查者因界面整洁而更易批准。
- 人类审查能力未随代理生成速度同步提升，差距扩大。

**审查前准备**
- 作者需在PR描述中清晰说明意图，避免代理生成冗长描述，并自行初审以示尊重。
- 审查者需理解代理缺乏上下文（如团队惯例、系统约束），依赖自身判断。

**五大危险信号与应对**
1. **CI规避**：检查是否降低覆盖率、跳过测试或修改工作流。任何弱化CI的变更均需一票否决。
2. **代码重复**：搜索新工具函数是否与现有代码重复，要求合并而非新增。
3. **幻觉正确性**：追踪关键路径的边界条件、权限检查和边缘案例，要求新增测试验证修复。
4. **代理失控**：对无计划或大型PR要求拆分，避免陷入无效循环。
5. **工作流安全**：审查LLM调用工作流，确保输入净化、权限最小化，禁止直接执行模型输出。

**审查流程建议（10分钟内）**
1. 分类任务（简单/复杂）。
2. 优先检查CI变更。
3. 扫描新工具函数并查重。
4. 追踪关键路径逻辑。
5. 验证安全边界。
6. 要求非平凡变更的测试证据。

**工具辅助**
- 先用GitHub Copilot等工具自动审查机械问题（如风格、基础错误），再聚焦人类判断部分。
- 可定制检查规则（如CI阈值、输入验证）以提升自动化效率。

**总结**
- 代理PR审查的核心是平衡效率与质量：让代理处理低阶任务，人类专注上下文判断。
- 坚持CI不妥协、关键路径追踪和结构化检查清单，以控制技术债务。
