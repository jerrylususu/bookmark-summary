# 一个 WebRTC 聊天室的三次演进
- URL: https://www.bmpi.dev/dev/free4chat/
- Added At: 2026-05-18 14:43:19
- Tags: #read #deepdive

## TL;DR
本文回顾了 free4chat 三次技术演进：从 Go+Pion 自建 SFU，到 Elixir+Membrane 实现集群与文字聊天，再到 Cloudflare 全栈实现运维归零。核心目标是在保证实时通信质量的同时降低运维成本，并探索 AI 融入。文章对比了不同技术栈的适用场景，并展望了 AI 与实时通信的未来方向。

## Summary
本文回顾了 free4chat 从 2022 年上线至今的三次技术演进，核心目标是探索如何在保证实时通信质量的同时，将运维成本降至最低，并尝试融入 AI 能力。

**WebRTC 基础与架构选择**
文章首先介绍了 WebRTC 的核心概念：信令与媒体分离、ICE 穿越 NAT、DTLS 安全层。在多人会议架构上，对比了 P2P、MCU 和 SFU，指出 SFU 是语音会议的最优解，因其服务器只转发不处理，成本低且延迟可控。

**第一版：Go + Pion，自建 SFU**
- **技术栈**：基于 Go 语言的 Pion 库和 Kraken SFU，部署在单台 AWS 服务器。
- **优势**：极度轻量，1 核 1G 内存支撑了近千人并发语音聊天，CPU 消耗极低。
- **局限**：单点故障，无集群扩容能力；存在内存泄漏问题；功能仅限语音，无文字聊天。
- **实测**：跨国通话稳定，验证了 SFU 在纯音频、低码率场景下的高效性。

**第二版：Elixir + Membrane，集群与文字聊天**
- **技术栈**：改用 Elixir 语言和 Membrane 多媒体框架，利用 Erlang/OTP 的原生集群能力。
- **优势**：开箱即用的集群支持，进程隔离和容错能力强；新增了文字聊天功能（Phoenix Channels）。
- **局限**：仍需自建 SFU 和 coturn 服务，运维成本中等。
- **实测**：集群稳定，调试体验优（LiveDashboard + remote REPL）。

**第三版：Cloudflare 全栈，运维归零**
- **技术栈**：完全依赖 Cloudflare 生态：Dyte（托管 SFU）、Turnstile（人机验证）、Workers KV（限流）、Durable Objects（AI 会话状态）、AI Gateway 和 Workers AI（模型推理）。
- **优势**：运维成本接近零，开发效率高（AI 辅助开发），边缘部署延迟低。
- **局限**：无法实现服务端音频注入（因托管 SFU 不暴露底层接口），AI 助手目前仅支持文字回复。
- **实测**：屏幕共享场景下线路稳定，验证了托管 SFU 的可靠性。

**横向对比与选择建议**
- **Go + Pion**：适合学习 WebRTC 底层、需要完全控制 SFU 的场景。
- **Elixir + Membrane**：适合生产级实时通信、高并发及未来需要 AI 语音注入的场景。
- **Cloudflare + RTK**：适合个人项目、快速验证、追求零运维的场景。

**AI 与实时通信的未来方向**
- **当前**：AI 助手参与文字聊天（如 Luna）。
- **近期**：语音 Bot（STT → LLM → TTS → 注入 SFU），但媒体注入仍是瓶颈。
- **远期**：AI 作为真正参与者，实时补充会议上下文、提供知识支撑，或用于语音陪伴场景。

**总结**
技术选型取决于边界匹配：Go + Pion 提供清晰的学习路径，Elixir + Membrane 展现优雅的分布式状态管理，Cloudflare 全栈则展示了运维归零的未来。AI 辅助开发显著提升了个人项目的可持续性。三个版本代码均在同一仓库的不同分支中。
